Claude Code loops, from single session to agent swarm

01 · Foundations

The mental model: one loop, one constraint

TL;DREverything Claude Code does is one loop — gather, act, verify, repeat — and every technique in this guide is a strategy for spending its scarcest resource: the context window.

Claude Code is not a chatbot with file access. It's an agentic harness around a model: it supplies tools, permission gating, memory loading, and the loop that chains actions together. When you hand it a task, it cycles through three blended phases until the work is done:

Phase A

Gather context

Search and read files, inspect git state, load CLAUDE.md, pull docs, ask you questions.

Phase B

Take action

Edit files, run commands, call MCP tools, commit — every action produces new evidence.

Phase C

Verify results

Run tests, builds, linters, screenshots. The result feeds the next iteration of the loop.

The loop adapts to the task: a codebase question may be pure gathering, a bug fix cycles through all three phases repeatedly, a refactor is verification-heavy. Claude chains dozens of actions, course-correcting on what each step returns — and you're part of the loop too, free to interrupt, redirect, or step away.

The governing constraint

The context window holds the entire conversation — every message, file read, and command output — and model performance degrades as it fills. A single debugging session can burn tens of thousands of tokens. Context is the scarcest resource you manage.

The autonomy ladder

Think of the four loop patterns as rungs on a ladder. Each rung trades more setup and more tokens for less of your attention:

Pattern I · §04cost 1×

Verification loops

One session that closes its own feedback loop with a check it can run. The default.
Pattern II · §05cost scales with N

Automation loops

Non-interactive claude -p runs in scripts, CI, and fan-out pipelines.
Pattern III · §06cost ~1–3×

Subagents

Disposable workers with isolated context, reporting summaries back to one session.
Pattern IV · §07cost ~N×+

Agent teams

Multiple full sessions with a shared task list and peer-to-peer messaging.

02 · Foundations

Anatomy of the loop

TL;DRModels reason, tools act, the harness glues them. Every extension mechanism plugs into a specific loop phase — pick where to intervene, and remember: CLAUDE.md is advisory, hooks are deterministic.

Under the hood the loop is a tight transaction: the model decides what to do next, the harness executes it, and the result becomes new evidence in the context window.

What Claude sees when a session starts

  • Your project — files in the directory and subdirectories, plus anything you grant access to.
  • Your terminal — any command you could run: build tools, git, package managers, scripts.
  • Your git state — current branch, uncommitted changes, recent history.
  • Your CLAUDE.md — persistent project instructions loaded every session.
  • Configured extensions — MCP servers, skills, hooks, subagent definitions.

Extension points map onto loop phases

The key architectural insight for advanced users: choosing an extension mechanism is choosing where in the loop to intervene.

MechanismLoop phaseNatureUse for
CLAUDE.mdGather · every sessionAdvisory, always loadedBroad conventions, commands Claude can't guess
SkillsGather · on demandAdvisory, loaded when relevantDomain knowledge, repeatable workflows
MCP serversGather + ActNew toolsIssue trackers, databases, design tools, monitoring
HooksAct + Verify gatesDeterministic scriptsThings that must happen every time, no exceptions
SubagentsWhole-loop delegationIsolated contextInvestigation, adversarial review
Agent teamsParallel loopsFull sessions + coordinationWork that benefits from peer coordination
Rule of thumb

A rule buried in a bloated CLAUDE.md gets ignored — it's advisory. Hooks always run. If a rule keeps being violated, don't add emphasis; convert it into a hook.

You are a component of the loop

The harness assumes an operator. Esc stops Claude mid-action with context preserved; Esc Esc / /rewind restores conversation, code, or both to any checkpoint. Every prompt creates a checkpoint and Claude snapshots files before each change — so "try something risky, rewind if it fails" is a legitimate strategy. (Checkpoints only track Claude's changes; they're not a git replacement.)

03 · Foundations

Context management

TL;DRTreat context like a performance budget. /clear between tasks, keep CLAUDE.md ruthlessly small, and after two failed corrections — reset and rewrite the prompt.

Senior engineers who get mediocre results from Claude Code almost always have a context problem, not a model problem. Everything you let into the window competes with your actual instructions.

The session-hygiene toolkit

CommandWhat it doesWhen to reach for it
/clearResets context entirelyBetween unrelated tasks — the single highest-leverage habit
/compact <instructions>Summarizes history with your guidance, e.g. /compact focus on the API changesLong sessions where the thread still matters
/rewind → summarizeCondenses messages from / up to a checkpointSurgical compaction of only part of the conversation
/btwSide question in a dismissible overlay — never enters historyQuick lookups that don't belong in the session
Status line · /contextContinuous visibility into context usageAlways-on; you can't manage what you can't see
--continue / --resume · /renamePersistent, named sessionsTreat sessions like branches — one workstream each
The two-correction rule

If you've corrected Claude twice on the same issue, the context is polluted with failed approaches — and every further correction competes with them. /clear and write a better initial prompt incorporating what you learned. A clean session with a sharper prompt almost always beats a long session with accumulated corrections.

CLAUDE.md: small, sharp, versioned

CLAUDE.md is loaded every session, which makes it expensive real estate. For each line ask: "would removing this cause mistakes?" If not, cut it.

✓ Belongs in CLAUDE.md
  • Build/test commands Claude can't guess
  • Style rules that differ from language defaults
  • Repo etiquette: branch naming, PR conventions
  • Architecture decisions specific to the project
  • Environment quirks and non-obvious gotchas
✗ Doesn't belong
  • Anything Claude can learn by reading the code
  • Standard conventions the model already knows
  • Detailed API docs — link instead
  • File-by-file codebase descriptions
  • "Write clean code" — self-evident filler

Check it into git so the team compounds it over time. Use @path/to/file imports for modular instructions, CLAUDE.local.md for personal overrides, and nested CLAUDE.md files in monorepos. Knowledge that's only sometimes relevant belongs in a skill, which loads on demand instead of taxing every session.

Steering compaction

Auto-compaction triggers as the window approaches its limit. You can shape what survives — add a rule to CLAUDE.md:

CLAUDE.md
When compacting, always preserve the full list of modified files,
the current task list, and any test commands used in this session.

Spec first, execute fresh

For larger features, split the loop across two sessions: an interview session that produces a spec, and a clean execution session that consumes it — starting with zero exploration noise.

prompt · interview session
> I want to build [brief description]. Interview me in detail using the
AskUserQuestion tool. Ask about technical implementation, UI/UX, edge
cases, and tradeoffs — skip the obvious questions, dig into the hard
parts I might not have considered. Keep going until we've covered
everything, then write a complete, self-contained spec to SPEC.md:
name the files and interfaces involved, state what's out of scope,
and end with an end-to-end verification step that proves it works.

Then /clear (or a fresh session) and execute: "Implement SPEC.md. Follow it exactly; where it's silent, match existing patterns in the codebase."

04 · Loop pattern I

Verification loops & TDD

TL;DRGive Claude a check it can run — tests, build, diff script, screenshot — and the loop closes itself. No check, no ship. Always demand the evidence, not the assertion.

Claude stops when the work looks done. Without a runnable check, "looks done" is the only signal — and you become the verification loop, catching every mistake by hand. Give Claude something that returns pass/fail and the loop closes itself: do the work, run the check, read the result, iterate until green.

The gating ladder

Once a check exists, decide how hard it gates the stop. Each step trades setup for attention:

LevelMechanismGate strengthBest for
1 · In-prompt"Run the tests and iterate until they pass"AdvisoryAny task, today, zero setup
2 · Session goal/goal condition — a separate evaluator re-checks after every turnPersistentMulti-turn work toward one outcome
3 · Stop hookYour script blocks the turn from ending until it passes (overridden after 8 consecutive blocks)DeterministicUnattended runs that must finish correctly
4 · Second opinionVerification subagent in fresh context tries to refute the resultAdversarialAnything you'd normally peer-review
Evidence, not assertions

Always ask Claude to show the proof — test output, the exact command and its result, a screenshot. Reviewing evidence is faster than re-running verification yourself, and it's the only way to trust sessions you weren't watching.

The strict TDD loop

TDD is the canonical verification loop: the failing test is the spec, and "tests pass" is an unambiguous exit condition. Two constraints matter: tests first, confirmed failing for the right reason — and implementation may not touch the tests.

prompt · TDD feature loop
> We're doing strict TDD for [behavior]. Expected input/output pairs:
[case 1 → result], [case 2 → result], [edge case → result].

Step 1: write failing tests covering these cases plus edge cases you
identify. Do NOT write any implementation. Run the tests and confirm
each fails for the right reason (missing behavior, not a typo).
Step 2: commit the tests.
Step 3: implement until all tests pass. You may not modify the tests.
Step 4: run the full suite, show me the output, then commit.
prompt · bug-reproduction loop
> Users report [symptom]. Likely area: [path/to/module], especially
[specific flow]. Write a failing test that reproduces the issue first,
then fix the root cause — do not suppress the error or special-case
the test. Run the full suite to check for regressions and show me
the output, including the new test passing.
prompt · visual verification loop (UI)
> [attach mock] Implement this design. When done, take a screenshot of
the running result, compare it to the mock, list every difference
(spacing, color, type, alignment), fix them, and re-screenshot.
Repeat until you can't find meaningful differences.

Writer / Reviewer: split the loop across sessions

A fresh context makes a better reviewer — Claude won't be biased toward code it just wrote. Implement in Session A, review in Session B, feed findings back to A. The same split works for tests: one session writes tests, another writes code to pass them — a division of labor that keeps either side from gaming the check.

Adversarial review, in-session

Before treating a task as done, have a subagent review the diff in fresh context. The bundled /code-review skill does a correctness pass; for plan conformance, write your own:

prompt · adversarial review subagent
> Use a subagent to review the [feature] diff against PLAN.md. Check
that every requirement is implemented, the listed edge cases have
tests, and nothing outside the task's scope changed. Report gaps
that affect correctness or the stated requirements only — not style
preferences.
Reviewer inflation

A reviewer prompted to find gaps will report some even when the work is sound — that's its job. Chasing every finding produces over-engineering: extra abstraction layers, defensive code, tests for impossible cases. Scope the reviewer to correctness and requirements; treat the rest as optional.

✓ Pros
  • Converts "watch the session" into "walk away from the session"
  • Exit condition is objective — no over- or under-execution
  • Cheapest pattern: one context window, no coordination overhead
  • TDD variant doubles as regression protection forever
✗ Cons
  • Only as good as the check — weak tests → confidently wrong code
  • Claude may satisfy the letter of a check (watch for test edits, suppressed errors)
  • Long fix-loops can cascade: a fix breaks another test, and so on
  • Verification output itself consumes context in long sessions

05 · Loop pattern II

Automation loops: headless, CI & fan-out

TL;DRclaude -p runs one prompt and exits — the bridge to scripts, CI, and cron. Fresh context per unit of work, but no human in the loop: invest in permissions, exit conditions, and calibration first.

claude -p "prompt" (print mode, formerly headless mode) executes a single prompt and exits — no interactive session. The Agent SDK is the programmatic TypeScript/Python equivalent.

shell · the three output modes
# One-off query, plain text
claude -p "Explain what this project does"

# Structured output for scripts
claude -p "List all API endpoints as JSON" --output-format json

# Streaming JSON for real-time processing (needs --verbose)
claude -p "Analyze this log file" --output-format stream-json --verbose

# Pipe data in, pipe results out
cat error.log | claude -p "Cluster these errors by root cause" \
  --output-format json | your_command

Scoping what an unattended loop can do

  • --allowedTools "Edit,Bash(npm test:*),Bash(git commit *)" — allowlist exactly the tools the task needs.
  • --permission-mode auto — a classifier reviews commands, blocking scope escalation and hostile-content-driven actions. In -p runs, auto mode aborts if the classifier repeatedly blocks — there's no human to fall back to.
  • Sandboxing (/sandbox) — OS-level filesystem and network isolation for defense in depth.
  • Prefer reversible actions; exclude destructive ones; define "done" explicitly so the loop doesn't over-execute.

The fan-out pattern

For large migrations or analyses, don't run one giant session — give each unit of work its own fresh, small context:

shell · fan-out migration
# 1. Have Claude generate the task list (one interactive session):
#    "List every file that still uses [old pattern] into targets.txt"

# 2. Loop with scoped permissions; test the prompt on 2-3 files first
for file in $(cat targets.txt); do
  claude -p "Migrate $file from [old pattern] to [new pattern].
    Follow the conventions in MIGRATION.md. Run the tests related
    to this file. Print exactly OK or FAIL: <one-line reason>." \
    --allowedTools "Edit,Bash(npm test:*),Bash(git commit *)" \
    >> migration.log
done

# 3. grep FAIL migration.log → triage the stragglers interactively
Calibrate before you scale

Run the loop on the first 2–3 files, read the output, refine the prompt, then unleash it on the full set. Prompt bugs at N=2,000 are expensive.

CI integration

shell · CI review gate
# In your pipeline, after checkout:
claude -p "Review the diff between origin/main and HEAD. Flag ONLY
  correctness, security, and data-loss risks — no style comments.
  Output JSON: {\"findings\": [{\"file\", \"line\", \"severity\", \"note\"}]}.
  Return an empty findings array if the diff is clean." \
  --output-format json --allowedTools "Read,Grep,Glob,Bash(git *)"

The same shape works for changelog generation, issue triage, labeling, or nightly dependency-risk scans. Use --verbose while developing the pipeline; turn it off in production.

The "Ralph" loop: while-true autonomy

A community pattern (not an official feature): run the same prompt in a bash loop, each iteration a fresh context, using the file system and git as the only memory. The prompt describes the goal, the state convention, and the exit condition; each run picks one unit of work, does it, commits, and exits.

shell · ralph-style loop (sandbox only)
# PROMPT.md contains: the goal, "read TODO.md, pick ONE item,
# implement it with tests, mark it done, commit" and the rule:
# "if TODO.md is empty, print DONE and stop."
while :; do
  out=$(claude -p "$(cat PROMPT.md)" \
    --allowedTools "Edit,Bash(npm test:*),Bash(git commit *)" \
    --output-format json)
  echo "$out" >> ralph.log
  grep -q '"DONE"' <<<"$out" && break
done
Handle with care

This trades all oversight for throughput. It only works when: the prompt is idempotent, each unit is independently verifiable (tests gate every commit), permissions are tightly scoped, it runs in a sandbox or throwaway container, and there's a hard exit condition plus an iteration cap. Never point it at anything with production access, and never combine it with permission-skipping flags outside isolation. Expect a large diff review at the end — budget for it.

✓ Pros
  • Fresh context per unit of work — no degradation at scale
  • Composable with everything: pipes, CI, cron, make
  • Massively parallelizable (fan-out across thousands of files)
  • Machine-readable output for downstream tooling
✗ Cons
  • No human in the loop — bad early decisions compound silently
  • Prompt quality is everything; iteration cycle is slower
  • Cross-file consistency is harder than in one session
  • Requires real investment in permissions, sandboxing, exit conditions

06 · Loop pattern III

Subagents: isolated context, condensed results

TL;DRDisposable workers with their own context window. The subagent reads fifty files; your session receives two paragraphs. Highest-leverage tool against context degradation.

A subagent is a separate, focused agent instance with its own context window that does one scoped unit of work and returns a condensed result to the main session. The main agent stays in charge of the narrative and the final decision; subagents are workers whose report comes back like a tool result. They can run in parallel, can be tool-restricted (e.g., read-only), and cannot talk to each other — only back to the parent. Built-ins include Explore, Plan, and general-purpose.

Use case 1 · Investigation without pollution

prompt · delegated research
> Use subagents to investigate how our authentication system handles
token refresh, and whether we have existing OAuth utilities I should
reuse. Report back: the relevant files, the current flow, and any
sharp edges — I don't need the file contents in this session.

Use case 2 · Parallel hypotheses

prompt · parallel debugging
> The p95 latency on [endpoint] doubled since last week. Spin up three
subagents in parallel: one investigating query patterns and indexes,
one profiling the middleware chain, one checking recent dependency
and config changes. Each reports its top suspect with evidence.
Then synthesize and tell me which hypothesis to test first.

Use case 3 · Custom specialists

Define reusable specialists in .claude/agents/ — frontmatter for name, description, allowed tools, and model, followed by the system prompt:

.claude/agents/security-reviewer.md
---
name: security-reviewer
description: Reviews code for security vulnerabilities
tools: Read, Grep, Glob, Bash
model: opus
---
You are a senior security engineer. Review code for:
- Injection vulnerabilities (SQL, XSS, command injection)
- Authentication and authorization flaws
- Secrets or credentials in code
- Insecure data handling
Provide specific line references and suggested fixes.

Invoke explicitly: "Use the security-reviewer subagent on this diff." Note the tool scoping — a read-only reviewer physically cannot "fix" things itself, which is exactly what you want from a reviewer.

✓ Pros
  • Protects the main session's context — the whole point
  • Fresh-context reviews are less biased than self-review
  • Parallelism inside one session, no coordination burden on you
  • Tool scoping reduces blast radius
✗ Cons
  • Each subagent re-pays initialization context — token cost multiplies
  • Fire-and-forget: no mid-task steering, no peer communication
  • Summaries are lossy; critical detail can be flattened
  • Doesn't inherit the parent's history — brief it fully

07 · Loop pattern IV

Agent teams: coordinated parallel sessions

TL;DRExperimental. Multiple full sessions — a lead, teammates, a shared task list, peer-to-peer messaging. The most capable pattern and the most expensive; use only when workers must react to each other.

Agent teams (experimental, disabled by default — enable with CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS in settings.json or your environment) coordinate multiple full Claude Code sessions. One session acts as team lead: it spawns teammates, breaks work into a shared task list, approves plans, and synthesizes results. Teammates each have their own context window, load the same project context (CLAUDE.md, MCP, skills) but not the lead's history, and — the key difference from subagents — message each other directly. You can also talk to any teammate yourself, without going through the lead.

ComponentRole
Team leadMain session: spawns teammates, assigns tasks, approves plans, synthesizes
TeammatesIndependent full sessions, each with its own context window
Task listShared work items teammates claim and complete (view with Ctrl+T)
MailboxPeer-to-peer messaging between any agents by name

Spawning a team

Describe the task and roles in natural language; name the teammates yourself if you want to address them later:

prompt · parallel refactor team
> Spawn three teammates for the checkout-module refactor: "api" owns
the API layer, "data" owns migrations, "tests" owns test coverage.
Coordinate through the shared task list; when api changes an
interface, it must message tests directly. Only approve plans that
include test coverage. No one merges without your review.
prompt · multi-angle design review
> I'm designing [system/feature]. Spawn three teammates to explore it
from different angles: one on developer experience, one on technical
architecture, one playing devil's advocate. Have them challenge each
other's findings over messages before you synthesize a
recommendation with the open disagreements listed explicitly.

Operator controls that matter

  • Delegate mode (Shift+Tab on the lead) — the lead stops implementing and only coordinates. Use it when the lead keeps "helping" with code.
  • Plan approval — require teammates to plan before touching code; steer the lead's judgment in your prompt ("reject plans that modify the DB schema without a migration").
  • Direct messaging — select a teammate in the agent panel (or its split pane) to redirect it, give it context the lead doesn't have, or interrupt with Escape.
  • Mixed models — run the lead on a stronger model for decomposition and plan review, teammates on a faster model for scoped implementation. A meaningful cost lever.
  • Worktrees — give teammates isolated git checkouts when they'd otherwise collide on the same files.
Token economics

Teams use significantly more tokens than a single session — every teammate is a full session with its own context window, so cost scales roughly linearly with team size (higher when teammates plan first). Worth it for research, review, and parallel feature work; wasteful for routine tasks. If workers don't genuinely need to coordinate, subagents or a fan-out script are cheaper.

✓ Pros
  • True parallel work with peer coordination — no round-trips through a parent
  • You can steer any individual worker mid-flight
  • Lead as quality gate: plan approval, sequencing, synthesis
  • Opposing-perspective teams surface disagreement a single agent smooths over
✗ Cons
  • Experimental: known limitations around resumption and shutdown
  • Highest token cost of any pattern
  • Permission prompts multiply across teammates — invest in your allowlist first
  • Coordination overhead only pays off on genuinely parallel work

08 · Practice

Choosing a loop: the decision matrix

TL;DRDefault to the cheapest pattern that fits. Escalate only when a specific limitation bites — most day-to-day work belongs in a single verification loop.
PatternContext isolationCoordinationCostYour attentionReach for it when…
Verification loop · §04None — one windowN/A1×Low–mediumAny task with a runnable check; the default
Subagents · §06Per-worker windows, summaries returnParent-only, fire-and-forget~1–3×LowInvestigation would flood your context; you want an unbiased reviewer
Automation / fan-out · §05Fresh context per invocationYour scriptScales with N, each unit smallSetup-heavy, then zeroMany independent, repetitive units; CI; scheduled work
Agent teams · §07Full window per teammateShared tasks + peer messaging~N× and upMedium (orchestration)Workers must react to each other's output; multi-angle analysis

Escalation triggers

  • Session keeps degrading mid-task despite /clear discipline → move investigation to subagents.
  • The task is "do X, N times, independently" → fan-out, never one long session.
  • Subagent workers keep needing each other's outputs and you're playing messenger → agent team.
  • The task would need 20+ coordinated steps unattended → reconsider scope first; a tighter spec beats a bigger loop.

And in the other direction: if an agent team's members never message each other, you paid team prices for subagent work.

09 · Practice

Failure patterns to recognize early

AP-01The kitchen-sink session

One task, then an unrelated question, then back to the first task. Context fills with irrelevant material that competes with your instructions.

Fix/clear between unrelated tasks; /btw for side questions.

AP-02Correcting over and over

Wrong → correct → still wrong → correct again. Every failed approach stays in context, actively steering the model back toward it.

FixTwo-correction rule: /clear, then a better prompt that encodes what you learned.

AP-03The over-specified CLAUDE.md

Important rules get lost in noise; Claude appears to "ignore instructions" when really the file is too long to weight properly.

FixRuthlessly prune. If a rule must always hold, make it a hook — hooks are deterministic.

AP-04The trust-then-verify gap

A plausible-looking implementation ships without a check, and the missing edge cases surface in production.

FixNo check, no ship. Tests, a build, a diff script, a screenshot — something must gate "done".

AP-05The infinite exploration

An unscoped "investigate this" sends Claude reading hundreds of files, torching the context before real work starts.

FixScope investigations narrowly, or delegate them to subagents so the reading happens elsewhere.

AP-06The unbounded autonomous loop

A headless loop with vague success conditions keeps "improving" forever, or cascades: a fix breaks a test, which triggers a fix, which breaks another.

FixMachine-checkable definition of done; iteration caps; checkpoint-and-continue between subtasks.

Meta-rule

These defaults aren't dogma. Sometimes context should accumulate (you're deep in one hard problem and the history is the asset). Sometimes a vague prompt is right (you want Claude's uninfluenced read). Notice what worked, notice what didn't — the loop between you and your own workflow is the one that compounds.

10 · Practice

Prompt library

Copy-paste starters, grouped by loop phase. Replace the brackets; keep the structure — each prompt encodes scope, sources, and a verification step on purpose.

Gather · exploration & onboarding

codebase questions — no special prompting needed
> How does logging work in this codebase?
> How do I add a new API endpoint here? Point me at the closest example.
> Look through [module]'s git history and summarize how its API came to be.
> What edge cases does [component] handle, and which are untested?

Plan · before the diff

plan mode · explore → plan → implement
# In plan mode (reads only, no changes):
> Read [dir] and understand how we handle [concern]. Also check how
we manage [related concern].
> I want to add [feature]. Which files change? What's the data flow?
Create a plan. Flag anything ambiguous in the requirements.
# Ctrl+G to edit the plan directly, then exit plan mode:
> Implement the plan. Write tests for [critical path], run the suite,
fix failures, and show me the final output.

Skip planning when the diff fits in one sentence — plan mode is overhead for typo-level changes.

Act · scoped implementation

pattern-following feature
> Look at how existing [widgets/handlers/jobs] are implemented —
[ExampleFile] is the reference pattern. Follow it to implement
[feature], using only libraries already in the codebase. Then run
[check command] and show the output.
root-cause build fix
> The build fails with this error: [paste error]. Fix it and verify
the build succeeds. Address the root cause — do not suppress the
error, loosen the config, or skip the failing step.

Verify · gates and reviews

hooks Claude writes for you
> Write a hook that runs [linter] after every file edit and reports
failures back into the session.
> Write a hook that blocks any write to the migrations folder.
> Write a Stop hook that runs [test command] and blocks ending the
turn until it exits 0.
fresh-context review
> Use a subagent to review this diff for edge cases, race conditions,
and consistency with our existing patterns. Only report findings
that affect correctness or the stated requirements.

Scale · headless one-liners

shell
# PR summary for a bot comment
git diff origin/main...HEAD | claude -p "Summarize this diff for a
reviewer: intent, risk areas, what to look at first. Markdown, short."

# Nightly triage
claude -p "Read the open issues with 'gh issue list'. Propose labels
and a priority for each. Output JSON." --output-format json \
  --allowedTools "Bash(gh issue list*)"

# Log forensics
cat prod-errors.log | claude -p "Cluster by root cause, give each
cluster a name, a count, and the single most representative line."
