PART 1

Vibe CodingGreat Demos, and Why They Collapse in a Real Repo

①The vibe ②The collapse ③The reframe

I open with the thing everyone in the room has already done, because starting anywhere else is dishonest. You paste a request into a chat window, code comes back, you paste it in, it runs. It feels like the future — and for a weekend toy it genuinely is. That's vibe coding: you describe, it generates, and you ride the vibes instead of reading the code. I don't mock it. Vibe coding is how most of us got convinced this technology is real, and for greenfield throwaways it's still the right speed.

Then the same trick meets a real repo, and it falls apart. The model doesn't know our project structure, so it invents one. It doesn't know our conventions, so it writes somebody else's. It can't run our tests, so "it works" means "it looked plausible." Every paste is a one-shot — no memory of the last attempt, no way to check itself — and the sixth round of "almost right" costs more than typing it yourself would have. The demo got 80% of the way there in five minutes; the last 20% eats the week.

The point I land: the failure isn't the model, it's the setup. A brilliant intern working through a mail slot — no repo access, no way to run anything, no memory between letters — would fail exactly the same way. Fixing the setup is the rest of this talk.

  • The collapse: invented structure, foreign conventions, "works" without verification

PART 2

AI EngineeringAn Agent in the Repo: You Steer, It Types

①The loop ②The proof ③Day to day ④The edges

The first real step up from the chat window: put the agent inside the repo. Claude Code reads the files, edits them, runs commands, runs the tests — and that changes the shape of the work from a one-shot into a loop: plan → act → observe → verify → repeat. You state intent, it acts, something observable happens — a test passes or fails, an error prints, a diff appears — and that observation steers the next turn. Your job stops being "produce the code" and becomes designing the loop and judging each turn. I give the loss its minute too — the traditional way of writing code, you and an editor and every line yours, had a real romance to it, and steering an agent isn't that. I'd rather name it than sell past it: the typing is what left the job; the deciding and judging is what grew. The hours tell the same story — they don't shrink so much as move downstream of the output: understanding what came back, verifying it, correcting it, fine-tuning it. And I put one correction on the table early: an AI is not a person and doesn't draw conclusions the way a human brain does — what it doesn't know it doesn't know, what you don't tell it to do it may skip, and what you don't tell it how to do it will absolutely do, its way. You feed it intent by the spoonful and treat it like the machine it is.

The proof is a build, not a claim — the answer to the "sure, but not on real code" objection. One vertical slice of the reference app — a save button, an API endpoint, persistence, a test that proves the round-trip — built entirely by steering the agent, no editor. The most useful minutes are the planted mistake: I under-specify on purpose, the agent guesses wrong, the test goes red, I feed the failure back, and the loop converges on its own instead of me fixing it by hand. Someone always asks about sharing records here; I say no sharing yet, on purpose — that debt comes due in Part 8.

I'm honest about the edges, because the skeptics in the back row are the ones I need: a one-character fix you can see is faster typed than described. If you can't verify the output and the cost of wrong is high, the agent's confidence is worth nothing. And an agent circling for three turns isn't a prompting problem — it's a context problem. Sometimes the context is missing; sometimes it's just full — quality slides as a session grows, a slope rather than a cliff, so I finish the slice before the dumb zone and hand long work to a fresh session with a written summary. Which is exactly the friction this level leaves you with: it works, and every single task starts with you typing the same orientation paragraph again.

Talk track

"You'll hear a folk number — 'it gets dumb around 100k tokens.' The honest version is worse for our intuition: there is no wall. The degradation is continuous, it starts in the low tens of thousands of tokens, and it keeps sliding. Chroma measured 18 frontier models in 2025 and quality dropped at every step up in length — even on trivial retrieval, even on million-token windows."

"Why? Attention gets spread thin — your one real instruction competes with thousands of tokens of noise. Position — models hold the start and end of a long context and quietly underweight the middle; 'lost in the middle' is a measured result, not a vibe. And poison — a coding session's context is a pile of failed attempts and stale file dumps, and the model keeps re-reading its own mistakes."

"The circling is the tell, and you can't talk a poisoned context back to smart. The move is structural: finish the slice before you're deep in the zone, /clear or /compact, and for genuinely long work hand a fresh session a written summary of where things stand. And stop spending the attention budget on garbage — giant logs and whole files it never needed."

Five do's, five don'ts

Do

  • Hand it a check it can run — a test, a build, a screenshot
  • Hit Esc the moment it goes sideways
  • /clear after two failed corrections — restart sharper
  • Point at a file — "copy this"
  • Read the diff like you'll defend it in review

Don't

  • Let "looks done" be the only signal
  • Politely wait out a bad run
  • Argue with a poisoned context
  • Describe in prose what a file already shows
  • Rubber-stamp because it looks plausible
Talk track

"Five habits, one per finger. Give it a check it can actually run — a test, a build, a screenshot — or 'looks done' becomes the finish line and you become the verification loop."

"When it goes sideways, interrupt — Esc mid-action, redirect. Don't politely watch a bad run finish. And after two failed corrections on the same thing, stop arguing: you just saw why — the context now carries every failed attempt, and that wreckage steers every next turn. /clear, fold what you learned into a sharper prompt, go again."

"Pointing at a file — 'do it like that one' — beats a paragraph of adjectives every time. And read the diff like you'll defend it in review. The dangerous approval isn't the first one, it's the tenth, when you're just clicking through."

PART 3

Agent-ReadyMaking the Repo the Agent Can Actually Work In

①Onboard ②Verify ③Our way ④Map & guard

By now everyone has felt the itch from Part 2: every new task opens with the same recited paragraph — what this project is, where the utils live, which design system we use, how to run the thing. So I pull back the curtain on that Part 2 build. It didn't go smoothly because the agent is magic — it went smoothly because the repo was set up for it. An agent starts every task cold: it knows none of what your teammates absorbed by osmosis — where things live, how you run them, what "done" looks like, what it must never touch. An agent-ready repo writes that tacit knowledge down and puts verification one command away. So this is the upstream lesson for everything after it: a context-starved loop spins, and the cure is almost always here, in the repo, not in a cleverer prompt.

I teach it as four moves, each one grounded in the reference app — ours first, then how you'd add it to a repo that has none.

  1. Give it an onboarding file — fix the cold start.The single highest-leverage move: a root-level instructions file the agent reads on every task — what this project is, how to run it, how to verify it, where the conventions live, and the hard guardrails. In Claude Code that file is CLAUDE.md (Cursor: .cursor/rules/*.mdc · Codex: AGENTS.md — same idea). Keep it short and true — a stale onboarding file is worse than none. And it layers: in a big repo or a monorepo, the root file carries the repo-wide rules and each package carries its own, so the agent reads only what the task touches.
    CLAUDE.md — the shape, not a novel
    # <Project> — what this is, in two lines
Run         npm run dev          # how to start it
Verify      npm run verify       # the one command that proves work → move 2
Conventions see docs/conventions.md   # link, don't restate
Guardrails  never commit secrets · don't hand-edit /migrations · ask before deleting data
  2. Make verification one command — give the loop something true to read.Part 2's loop only closes on a real observation. The repo's job is to make that observation cheap and discoverable — one command any human or agent can run. You don't need tests to start: the floor is "does it compile, does it lint, does it run" — even that gives the loop a real verdict to read instead of the agent's say-so; the ceiling is a test that proves the behavior round-trips. Wire whatever you have behind one obvious script and name it in the onboarding file.
    one discoverable command — floor today, ceiling later
    // package.json → "scripts"
"verify": "tsc --noEmit && eslint . && npm run build"   // floor: compiles · lints · builds — cheapest "does it run"
//          … && vitest run   once you have tests          // ceiling
  3. Write down "our way" — and leave one example to copy.The patterns you'd correct in code review are the ones the agent can't guess. Put them in a short conventions doc — but more powerful than any prose, leave one golden-path example in the repo to pattern-match against (our add-endpoint-our-way reference is exactly this). "Do it like that file" beats a page of rules.
  4. Map the territory and set the guardrails.A few lines on where things live and where the entry points are, so the agent isn't spelunking — then the off-limits list: secrets handling, generated files, the paths it must not edit. Guardrails are how you let the agent move fast without holding your breath.

One line I hold firm: this is the passive layer — the always-on context any agent reads cold. Packaging a repeated move into a reusable skill is the active layer, and that's Part 4 — don't conflate them. First we lay the ground the agent stands on; then we teach it your moves.

Then the payoff, before and after: extend the CLAUDE.md, rerun the same kind of loop from Part 2, and the difference good context makes shows itself — the agent stops guessing field names and starts following the house style.

PART 5

The HarnessWhat Turns a Model into an Agent

①Text in, out ②The harness ③The parts ④The product

Time to strip the magic, because everything before this part quietly depended on it. A language model can't do anything. It reads text and it produces text — that's the entire interface. It cannot open a file, run a test, click a button, or remember yesterday. So how did the thing in Part 2 edit our repo and run our suite? It didn't. The harness did — the program wrapped around the model that turns its text into actions and the results back into text. When the model writes "run the tests," the harness actually runs them and feeds the output back in. Take the harness away and your "agent" is a very articulate person in a locked room, passing notes under the door.

Once the room sees that, the vocabulary stops being buzzwords and becomes parts of a machine. Tools are the verbs the harness exposes — read a file, edit it, run a command; the model requests, the harness executes. Hooks are your code intercepting its moves — a script that runs before every commit, a rule that blocks any command touching prod. Guardrails are the permission line: what it may do freely, what needs your yes, what it may never do. And subagents are the harness spawning a fresh agent with a clean context — the mechanism that makes the checker patterns in Part 8 possible at all. The loop from Part 2 doesn't live in the model either; the model just talks. The harness is what loops.

Then the landing, which reorders how people argue about tooling: Claude Code, Cursor, and Codex run the same handful of models underneath. The product — the thing you're actually choosing, the thing that decides what your agent can and can't do — is the harness. And once the model is just a component, it becomes a per-task knob: heavy reasoning — the architecture call, the two-day bug — gets the big, slow, expensive one; mechanical bulk — renames, boilerplate, sweeps — gets the fast, cheap one, and subagents can run the cheap model while the main loop stays smart. There's a second dial on the same panel: effort — how hard the model thinks on a given task — dialed down for the sweep, up for the two-day bug, and it moves latency and spend, not just quality. Every "which AI tool" debate is a harness debate wearing a model costume.

PART 7

①Failure modes ②The rule

By this point the room has met both, and I stop to draw the line hard, because these two get mixed up on every team I've coached — someone writes a skill hoping the agent grows a new sense, or wires an MCP server hoping it teaches house style. Both fail, each in its own way, and the fix is one distinction. Autonomy amplifies — it doesn't fix

an unverifiable condition is vibe coding on a timer — "make the dashboard feel snappier"
the evaluator reads only the transcript — a coverage number the agent never printed doesn't exist
unattended turns spend real tokens — bound them: "or stop after 20 turns"

Talk track
