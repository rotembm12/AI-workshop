PART 1

Vibe CodingGreat Demos, and Why They Collapse in a Real Repo

I open with the thing everyone in the room has already done, because starting anywhere else is dishonest. You paste a request into a chat window, code comes back, you paste it in, it runs. It feels like the future — and for a weekend toy it genuinely is. That's vibe coding: you describe, it generates, and you ride the vibes instead of reading the code. I don't mock it. Vibe coding is how most of us got convinced this technology is real, and for greenfield throwaways it's still the right speed.

Then I bring it into a real repo and let the room watch it fall apart. The model doesn't know our project structure, so it invents one. It doesn't know our conventions, so it writes somebody else's. It can't run our tests, so "it works" means "it looked plausible." Every paste is a one-shot — no memory of the last attempt, no way to check itself — and the sixth round of "almost right" costs more than typing it yourself would have. The demo got 80% of the way there in five minutes; the last 20% eats the week.

The point I land: the failure isn't the model, it's the setup. A brilliant intern working through a mail slot — no repo access, no way to run anything, no memory between letters — would fail exactly the same way. Fixing the setup is the rest of this talk.

What I cover
  • What vibe coding actually is — describe, generate, don't read — and where it legitimately wins
  • The live collapse: invented structure, foreign conventions, "works" without verification
  • Why the last 20% eats the week — one-shots don't converge
  • The reframe: the model isn't the bottleneck, the setup is
PART 2

AI EngineeringAn Agent in the Repo: You Steer, It Types

The first real step up from the chat window: put the agent inside the repo. Claude Code reads the files, edits them, runs commands, runs the tests — and that changes the shape of the work from a one-shot into a loop: plan → act → observe → verify → repeat. You state intent, it acts, something observable happens — a test passes or fails, an error prints, a diff appears — and that observation steers the next turn. Your job stops being "produce the code" and becomes designing the loop and judging each turn.

I prove it live, because no slide kills the "not on real code" objection. One vertical slice of the reference app — a save button, an API endpoint, persistence, a test that proves the round-trip — built entirely by steering the agent, no editor. The most useful minutes are the planted mistake: I under-specify on purpose, the agent guesses wrong, the test goes red, I feed the failure back, and the room watches the loop converge instead of watching me fix it by hand. Someone always asks about sharing records while I'm in there; I say no sharing yet, on purpose — that debt comes due in Part 8.

I'm honest about the edges, because the skeptics in the back row are the ones I need: a one-character fix you can see is faster typed than described. If you can't verify the output and the cost of wrong is high, the agent's confidence is worth nothing. And an agent circling for three turns isn't a prompting problem — it's a context problem. Which is exactly the friction this level leaves you with: it works, and every single task starts with you typing the same orientation paragraph again.

What I cover
  • The agentic loop — plan → act → observe → verify — and why iteration plus verification beats one-shotting
  • The live proof: a tested vertical slice through prose alone, planted mistake and recovery included
  • Reading agent output critically — the test result is the truth, not the agent's confidence
  • When the loop is the wrong tool, said plainly — and the repeated-context itch that sets up Part 3
PART 3

Agent-ReadyMaking the Repo the Agent Can Actually Work In

By now everyone has felt the itch from Part 2: every new task opens with the same recited paragraph — what this project is, where the utils live, which design system we use, how to run the thing. So I pull back the curtain on that live build. It didn't go smoothly because the agent is magic — it went smoothly because the repo was set up for it. An agent starts every task cold: it knows none of what your teammates absorbed by osmosis — where things live, how you run them, what "done" looks like, what it must never touch. An agent-ready repo writes that tacit knowledge down and puts verification one command away. So this is the upstream lesson for everything after it: a context-starved loop spins, and the cure is almost always here, in the repo, not in a cleverer prompt.

I teach it as four moves and do each one live on the reference app — showing ours, then how you'd add it to a repo that has none.

  1. Give it an onboarding file — fix the cold start.The single highest-leverage move: a root-level instructions file the agent reads on every task — what this project is, how to run it, how to verify it, where the conventions live, and the hard guardrails. In Claude Code that file is CLAUDE.md (Cursor: .cursor/rules/*.mdc · Codex: AGENTS.md — same idea). Keep it short and true — a stale onboarding file is worse than none.
    CLAUDE.md — the shape, not a novel
    # <Project> — what this is, in two lines
Run         npm run dev          # how to start it
Verify      npm run verify       # the one command that proves work → move 2
Conventions see docs/conventions.md   # link, don't restate
Guardrails  never commit secrets · don't hand-edit /migrations · ask before deleting data
  2. Make verification one command — give the loop something true to read.Part 2's loop only closes on a real observation. The repo's job is to make that observation cheap and discoverable — one command any human or agent can run. You don't need tests to start: the floor is "does it compile, does it lint, does it run" — even that gives the loop a real verdict to read instead of the agent's say-so; the ceiling is a test that proves the behavior round-trips. Wire whatever you have behind one obvious script and name it in the onboarding file.
    one discoverable command — floor today, ceiling later
    // package.json → "scripts"
"verify": "tsc --noEmit && eslint . && npm run build"   // floor: compiles · lints · builds — cheapest "does it run"
//          … && vitest run   once you have tests          // ceiling
  3. Write down "our way" — and leave one example to copy.The patterns you'd correct in code review are the ones the agent can't guess. Put them in a short conventions doc — but more powerful than any prose, leave one golden-path example in the repo to pattern-match against (our add-endpoint-our-way reference is exactly this). "Do it like that file" beats a page of rules.
  4. Map the territory and set the guardrails.A few lines on where things live and where the entry points are, so the agent isn't spelunking — then the off-limits list: secrets handling, generated files, the paths it must not edit. Guardrails are how you let the agent move fast without holding your breath.

One line I hold firm: this is the passive layer — the always-on context any agent reads cold. Packaging a repeated move into a reusable skill is the active layer, and that's Part 4 — don't conflate them. First we lay the ground the agent stands on; then we teach it your moves.

Then I make it real in ninety seconds: I add or extend the CLAUDE.md live, rerun the same kind of loop from Part 2, and let the room watch the difference good context makes — the agent stops guessing field names and starts following the house style.

What I cover
  • Why an agent starts every task cold — and what "agent-ready" actually means
  • The onboarding file (CLAUDE.md / AGENTS.md): what goes in it, and why short-and-true beats long-and-stale
  • Verification as a property of the repo — one command, floor (compiles/lints/runs) to ceiling (tests)
  • Conventions plus a golden-path example to copy; a territory map and hard guardrails
  • The line between this passive substrate and the active skills layer in Part 4

① Trigger · how it's invoked

Two ways in — pick on purpose

User-invoked

  • Only you fire it, by name
  • No description in the agent's context
  • Cost: cognitive load — you must remember it exists

Model-invoked

  • The agent fires it itself (and you can too)
  • Description sits in context every turn
  • Cost: context load + it may not fire when it should

Our endpoint skill → model-invoked. We want the agent to reach for it automatically. So the description has to earn its keep.

PART 5

The HarnessWhat Turns a Model into an Agent

Time to strip the magic, because everything before this part quietly depended on it. A language model can't do anything. It reads text and it produces text — that's the entire interface. It cannot open a file, run a test, click a button, or remember yesterday. So how did the thing in Part 2 edit our repo and run our suite? It didn't. The harness did — the program wrapped around the model that turns its text into actions and the results back into text. When the model writes "run the tests," the harness actually runs them and feeds the output back in. Take the harness away and your "agent" is a very articulate person in a locked room, passing notes under the door.

Once the room sees that, the vocabulary stops being buzzwords and becomes parts of a machine. Tools are the verbs the harness exposes — read a file, edit it, run a command; the model requests, the harness executes. Hooks are your code intercepting its moves — a script that runs before every commit, a rule that blocks any command touching prod. Guardrails are the permission line: what it may do freely, what needs your yes, what it may never do. And subagents are the harness spawning a fresh agent with a clean context — the mechanism that makes the checker patterns in Part 8 possible at all. The loop from Part 2 doesn't live in the model either; the model just talks. The harness is what loops.

Then the landing, which reorders how people argue about tooling: Claude Code, Cursor, and Codex run the same handful of models underneath. The product — the thing you're actually choosing, the thing that decides what your agent can and can't do — is the harness. Every "which AI tool" debate is a harness debate wearing a model costume.

What I cover
  • Text in, text out — what a model actually is, and why it can't act alone
  • Tools: the verbs the harness executes on the model's behalf
  • Hooks and guardrails: your code and your permission lines around its moves
  • Subagents: fresh contexts on demand — the mechanism behind maker/checker
  • Same models, different harnesses — why the harness is the product

①Capability ②Connect ③Curate ④Cost

Part 5 left the room with a harness whose tools are all the agent has — and smart people didn't sit still with that. MCP is how you expand what an agent can do without building the ability from scratch for your specific harness: not new knowledge — that was Part 3's context and Part 4's skills — but new capability. A skill is a file you write; an MCP is a server you connect. Open a browser, generate an image or a video, query Snowflake or BigQuery, search your Sentry logs — thousands more, and the same server works in every harness that speaks the protocol.

The token economics matter and I say them plainly: the logic runs on the provider's side, so what enters your context is a tool call and its result — not the machinery behind it. PART 7

Skills vs MCPTeach It a Move, or Plug In a Capability?

By this point the room has met both, and I stop to draw the line hard, because these two get mixed up on every team I've coached — someone writes a skill hoping the agent grows a new sense, or wires an MCP server hoping it teaches house style. Both fail, each in its own way, and the fix is one distinction. A skill is knowledge: markdown in your repo that teaches a move the agent could already make but makes wrong, or differently every time — your endpoint structure, your doc format, your review checklist. An MCP is capability: a server you connect that gives it verbs it didn't have at all — a browser, a warehouse, your prod logs. A skill will never give the agent eyes; an MCP will never give it taste.

The costs differ the same way, and cost is where teams actually get burned. A skill is yours — plain text, versioned with the code, reviewed in a PR, and it costs context only when it fires. An MCP is someone else's code running with your access — you vet it like a dependency, and its tool definitions sit in every prompt whether today's task needs them or not. Cheap to write versus cheap to run; trust yourself versus trust a vendor.

The rule I leave them with fits in one breath: could the agent already do it, just badly? Write a skill. Can it not do it at all? Wire an MCP. And the two compose — the best skills orchestrate MCP tools: "check the error budget" is a skill that knows your thresholds and calls the Sentry server to read the truth.

  • Knowledge vs capability — where each one lives, what each one costs, who you trust
  • The one-breath rule: does it badly → skill · can't do it at all → MCP
  • Composition: skills that orchestrate MCP tools

①Grill ②Spec ③Verify ④Architect ⑤Slice ⑥Build

Back in Part 2's live build I said it out loud: no sharing yet. This is where that deferral comes due. One fuzzy sentence — "let people share their records" — goes in; merged, reviewed, tested code comes out, and the room watches every stage in between. The disciplines are all behind us by now; this part chains them into one pipeline on the reference app — the flagship block of the talk.

One idea carries the whole ninety minutes: produce in one context, verify in a clean one. The agent that argued the spec into shape is the worst judge of whether the spec is complete — it heard the argument, so it shares the blind spots. At every stage a fresh context that never saw the drafting checks the work: a gaps report on the PRD, a grilling pass on the architecture, a reviewer on the code. Nobody verifies their own homework, the AI included.

The pipeline descends by altitude — WHY → WHAT → prove it → WHERE → HOW MUCH → DO. Grill until "share" means one specific thing. Write the spec with the agent. Hand it to a clean context and loop until the gaps report comes back empty. Let the agent propose where the change should live, and judge the candidates. Cut the spec into slices an agent can finish. Then build and review in separate contexts, and merge.

Delivery is follow-along: I drive the two divergent beats — the grilling and the architecture — and the room runs the crisp ones on their own machines. At every stage boundary a frozen checkpoint file drops and everyone re-syncs from it, so a wandering agent never strands anyone. The same files go home as the worked reference.

The honest caveat — this is the heavy-feature pipeline
Skip it for one-liners and spikes.A rename, a one-character fix, a tracer you'll delete tomorrow — running six stages on those is ceremony, not planning. Run the loop and move on.
Run it when the feature is big, fuzzy, or shared.Big enough to slice, fuzzy enough that the agent will drift, or shared enough that someone else needs the same picture you carry in your head — that's the feature this pipeline is for.
What I cover
  • The maker/checker engine — produce in one context, verify in a clean one, at every altitude
  • Grilling a fuzzy one-liner into a grounded goal (/grilling) — the agent interrogates from the code's reality, then synthesizes
  • The six-pillar PRD written with the agent (write-a-prd) — stories that enumerate, non-goals that carry the weight, the test seam picked up front

① Grill · WHY — I drive, you watch

No building until "share" means one specific thing

public link, or private by default?
view-only, or can a viewer edit?
anyone with the link, or invited accounts?
owner keeps editing — do viewers see it live? revoke? expiry?
"share with anyone" + "only I can edit" — so what does the link actually do?

Not a blank-form interview — the agent has read the repo, knows there's an ownerId, and grills from the code's reality. Then it synthesizes. Checkpoint → grounded-context.md. Command: /grilling.

Talk track

"I hand the agent the dev's real one-liner — 'let people share their records' — and I let it interrogate me. It's read the code; it grills from what's actually there, not from a form."

"You said 'share with anyone' and 'only I can edit' — so is the link view-only? What about a co-owner on the same plan? That contradiction is the moment 'share' stops being one word."

do Dwell on the contradiction — it's the first moment to protect. Drop grounded-context.md at the gate.

next ▸ ② spec · your hands

② Spec · WHAT — your hands now

Six pillars, written with the agent

Problem from the user's side, not the implementation's Solution behavior a user can see User stories a long numbered list — enumerate, don't gesture Implementation the seam: canView / canEdit — created on purpose Testing every share story verified at that one seam Out of scope NOT real-time collab · NOT public discovery · NOT comments

Non-goals carry the weight — dwell there, rush the headings. Checkpoint → share-record.prd.md. Command: write-a-prd.

Talk track

"Stories enumerate the real cases: 'as an owner I share a view-only link so a friend sees my plan without editing it.' 'As an owner I revoke a link so access ends.' A long list is the point — gesturing is how drift gets in."

"And I pick the test seam before any code exists: the access decision — canView, canEdit. Highest seam, fewest seams; it doesn't exist yet, so we create exactly one. Every share story gets verified right there."

do A five-minute synthesis live, not a form to fill. At the gate everyone adopts the canonical frozen PRD, so the next beat runs against one spec.

next ▸ ③ verify the plan

③ Verify · a context that never heard the argument

Loop until the gaps report comes back empty

Open a clean context. Hand it only the PRD. One ask: find the gaps and contradictions.

clean context · gaps report
Revocable links — but nothing says what a revoked link shows the visitor. Expiry is in scope with no story for an expired link. "View-only" and "collaborator" both listed, but Out-of-Scope says no editing — which is it?
Talk track

"This agent never heard our argument and doesn't know how hard we worked. It reads the spec cold — and that's exactly why its report is worth something."

"Take the report back to the plan agent — it has the context to fix each hole — then run the checker again. The payoff is the empty second report. Keep this move in your pocket; it comes back at build time."

do Room runs this hands-on from the frozen PRD. Checkpoint → tightened PRD + gaps-report.md. Say it out loud: this is a discipline, not a tool — a second window is enough.

next ▸ ④ architect · where

④ Architect · WHERE — I drive; the new material

One access seam instead of scattered checks

Before · shallow

  • "is this mine?" = record.ownerId === userId
  • copy-pasted across every handler
  • add sharing → the leak spreads to every endpoint

After · deep

  • one access module: canView / canEdit
  • endpoints ask the policy — they don't re-derive it
  • the policy is tested once, not per handler

The agent explores the repo and renders an HTML candidate report — per approach: files · problem · solution in plain English · benefits in locality and leverage · a before/after diagram · a badge: Strong / Worth exploring / Speculative. Then a grilling loop down to the winner. Checkpoint → architecture.md.

Talk track · give it air

"Nothing else in the day teaches this. The report names the friction in today's shape: 'is this mine?' is a scattered ownership check, copy-pasted across handlers — add sharing and the leak spreads everywhere."

"The winning candidate collapses it behind one seam. Locality: all the rules in one module. Leverage: every endpoint benefits at once, and the policy is tested once. That's the vocabulary for judging any candidate."

do The before/after drawing is the keeper visual. Keep the rejected candidates in architecture.md — with why. Commands: codebase-design + design-an-interface ("design it twice, in parallel"). Trim lever: 10 min — show the winner, skip the losers.

next ▸ ⑤ slice · how much

⑤ Slice · HOW MUCH — your hands

Slice 1 is ugly, end-to-end, and works

slice 1 read-only share link — token → shared view → a test proves a non-owner can view via link and cannot edit ← ugly · end-to-end · works slice 2 revoke + expiry blocked by 1 slice 3 invited collaborators with edit blocked by 1

Each slice cuts through every layer — never a horizontal "build the data layer" task. Acceptance criteria are written against the access seam from ④. Checkpoint → slices/. Command: prd-to-issues (or prd-to-plan for local markdown).

Talk track

"Slice 1 is a raw, unstyled link that genuinely round-trips. Later slices add width, not depth. Engineers trained on tidy architecture resist the ugliness — that resistance is the thing to talk through."

"Each issue is independently grabbable: context gathered, acceptance criteria at the seam, blocked-by marked — 2 and 3 wait on 1's token and policy. Publish to whatever tracker your team uses."

watch for horizontal slicers as the room cuts from the frozen PRD + architecture.md — it's the failure mode here.

next ▸ ⑥ build + review

⑥ Build + Review · DO — the relay

One context builds. A second one judges.

build a clean context takes slice 1 + architecture.md + the card review a second clean context — conventions · criteria · tests green → findings refactor a third pass works the findings merge slice 1 lands

The knobs I name and don't demo: AFK vs attended · one slice vs fan-out parallel · TDD vs not. Today: attended, slice 1 only, full relay visible. Command: your coding agent, then /code-review — it already reviews on Standards + Spec.

Talk track · the punchline

"The reviewer never saw the builder's rationalizations — and it catches that the shared-view endpoint renders an expired link, because the builder forgot to ask canView. That catch is the whole part's punchline. Land it."

"In real life you fan slices 2 and 3 out AFK overnight and review them in the morning. Today we stay attended and watch the relay end to end."

do Room builds hands-on; merge slice 1 at the gate. If time is tight, drop the TDD aside — the relay is the lesson.
